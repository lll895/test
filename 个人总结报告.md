## 一、项目概述

本项目是一个基于 RAG（Retrieval-Augmented Generation）架构的企业内部知识库问答系统。采用 Vue 3 + Element Plus 作为前端框架，Flask + SQLAlchemy 作为后端框架，LangChain + Chroma + Ollama 实现文档向量化和智能检索问答。

从最初的 v2.0 基础版本出发，经过三天的迭代开发，完成了 CI/CD 工程化建设、单元测试覆盖、密码重置、文档版本管理、在线编辑、前端 UI 全面升级、权限控制等功能，最终形成一个具备生产级别工程化水平的企业级应用。

## 二、遇到的最大困难及解决过程

### 困难一：数据库版本兼容性引发的连锁故障

**问题描述**：
在添加文档版本管理功能时，我在 Document 模型中新增了 `version`、`version_group_id`、`change_note` 三个字段，同时为密码重置功能在 User 模型中添加了 `reset_token` 和 `reset_token_expires` 字段。这些改动在 SQLite 测试环境中一切正常，但部署到 MySQL 生产数据库后，因为数据表结构没有同步更新，导致**整个系统崩溃**——知识文档页、搜索、收藏夹、对话历史等几乎所有页面都报 500 错误，错误信息全部指向 `Unknown column 'xxx' in 'field list'`。

**解决过程**：
1. **定位问题**：通过查看 Flask 后端的错误日志，发现所有 SQL 查询都在尝试访问不存在的列。根源是我在 Python 模型中添加了新字段，但 MySQL 实际表结构没有对应的列。
2. **临时止血**：移除 User 模型上的 `reset_token` 字段，改用独立的 `PasswordResetToken` 表。因为 User 模型被大量其他模型通过外键关联引用，修改它会导致连锁的 JOIN 查询失败。
3. **根本解决**：编写了自动迁移脚本 `scripts/migrate.py`，它通过 `SHOW COLUMNS` 检查字段是否存在，只在缺失时执行 `ALTER TABLE`，不依赖任何 ORM 功能。

**反思**：
这个问题的教训是：**在开发阶段就应该引入数据库迁移工具**（如 Alembic/Flask-Migrate），而不是依赖 `db.create_all()`。`create_all()` 只建表不改表，在迭代开发中几乎必然导致生产环境的一致性问题。另外，修改被广泛引用的模型（如 User）时要格外小心，一个字段的添加可能通过 ORM 的关联关系影响到十几个看似不相关的查询。

### 困难二：AI 辅助下的登录重定向循环问题

**问题描述**：
在前端 UI 美化过程中，路由配置被修改后，普通用户登录成功后会立刻被踢回登录页，显示"登录已失效"。这是一个典型的、无法通过代码审查直接发现的运行时逻辑错误。

**解决过程**：
1. **现象分析**：用户能登录成功（API 正常返回 token），但页面加载后立即跳回登录页。说明问题不在登录 API，而在前端路由或页面渲染。
2. **逐步排查**：检查了 token 存储、路由守卫、页面生命周期等多个环节。最终发现根路由 `/` 固定重定向到 `/home`，而 `/home` 设定了 `roles: ['admin']`。当普通用户因任何原因被重定向到 `/` → `/home` → 角色检查失败 → `next('/')` → 无限循环。循环中某个时刻 token 被清除，显示"登录失效"。
3. **修复**：将根路由的 `redirect: '/home'` 改为函数形式，根据 `user.role` 动态决定跳转目标；角色检查失败时不再跳到 `/`，而是跳转到对应的用户首页。

**反思**：
这个 bug 的根因是**路由守卫和重定向之间形成了一个正反馈环**。在复杂的前端路由配置中，这样的循环很隐蔽——代码单独看每一段都是"正确的"，但组合起来就出问题。这让我意识到：**AI 生成代码后，不能只看单段代码的正确性，还要关注代码之间的交互逻辑**。

## 三、对 AI 辅助编程的心得体会

### 3.1 AI 的优势

**效率提升极其显著**。在这三天的开发中，从 CI/CD 配置、单元测试编写、完整的前端页面开发到数据库迁移脚本，工作量保守估计在 3000 行以上。如果用传统方式手写，至少需要两周时间。AI 将这些工作的耗时压缩到了以小时为单位。

**多语言无缝切换**。项目同时涉及 Python（Flask）、JavaScript（Vue 3）、CSS、Shell 脚本、YAML（GitHub Actions）、Dockerfile、Markdown 等多种语言/格式。AI 可以在这些语言之间自由切换，无需我手动切换上下文——这在传统开发中是一个很大的认知负担。

**测试驱动开发的真正落地**。在写后端测试时，我只需要描述"测试登录成功的场景"，AI 就能自动生成包含 setup、执行、断言三个阶段的完整测试代码，并且覆盖成功路径和异常路径。这让我这个本来不太爱写测试的人，最终交付了 172 个测试用例。

### 3.2 AI 的局限

**对运行时行为的理解有限**。上述的路由循环问题就是一个典型例子——AI 可以生成正确的路由配置代码，但它无法"运行"这段代码来发现循环依赖。静态代码审查和动态运行时调试，仍然需要人类开发者的参与。

**上下文窗口的约束**。在一次对话中，AI 能记住的信息是有限的。当项目增长到 70+ 文件后，AI 有时会忘记之前的架构决策，导致前后不一致。比如在数据库迁移的问题上，AI 一开始在 User 模型上直接加字段，但后来才意识到这会破坏所有 JOIN 查询。

**安全意识的盲区**。AI 默认生成的代码更关注"功能正确"，而不是"安全正确"。代码审查中发现的 MD5 密码、硬编码密钥等问题，都是 AI 默认生成的。这提醒我：**AI 生成的代码必须经过安全审查才能用于生产环境**。

### 3.3 与 AI 协作的最佳实践

通过这个项目，我总结出几条和 AI 协作开发的有效方式：

1. **分阶段提问**：不要一次性提所有需求，而是按"基础功能 → 问题修复 → 优化改进"的节奏分阶段进行，每个阶段聚焦一个目标。
2. **提供上下文**：在提问时附上相关文件的路径和关键代码片段，AI 的回答准确率会显著提升。
3. **验证 + 追问**：AI 第一次给出的方案不一定最优，需要实际运行验证，发现问题后继续追问。这次项目中的很多 bug 都是在"验证-反馈-修复"的循环中解决的。
4. **异步执行复杂任务**：对于大型重构（如全面美化 11 个页面），让 AI 在后台并行执行，可以大大提高效率。

## 五、总结与展望

这三天的高强度开发，让我深刻体验了 AI 辅助编程的生产力。一个原本需要团队协作数周的项目，在 AI 的帮助下由单人三天完成，这在以前是不可想象的。

同时我也认识到：**AI 不是替代开发者，而是放大开发者的能力**。代码生成只是第一步，架构决策、安全审查、bug 调试、系统集成这些"高价值工作"仍然需要人类开发者来完成。未来的人机协作模式，更像是"指挥官与士兵"的关系——人类指明方向、做出决策，AI 负责执行和反馈。

对于这个项目，后续可以继续优化的方向：
1. 引入 Alembic 管理数据库迁移，彻底解决字段版本问题
2. 将 MD5 密码升级为 bcrypt，修复已知安全漏洞
3. 添加更多集成测试，覆盖流式问答等复杂场景
4. 接入真正的邮件服务，完成密码重置的生产化流程
